This page documents the backend exactly as the current code behaves. It is meant for frontend and support teams: each route includes what it does, which files control it, expected request shape, actual response shape, auth requirement, and what specific errors mean.

Important: the project does not use a single response envelope everywhere. The frontend should read the route docs for the exact shape.

Primary success envelope used by many controllers

{
  "status": 200,
  "message": "some message",
  "data": {}
}

Implemented by utils/errorHandeler.js through handelResponse().

Auth middleware and a few controllers use a different shape

{
  "success": true,
  "message": "User verified",
  "user": {
    "id": 42,
    "email": "buyer@example.com"
  }
}

Seen in middlewares/clientAuth.js, services/verifyUserLogin.js, and services/logout.js.

Global error handler envelope

{
  "status": "fail",
  "message": "Invalid Google ID token."
}

Returned by middlewares/error_handeler.js for thrown errors, especially AppError and unknown server failures.

Error Model

Status What it means in this project
400 Bad input, missing required field, invalid type, invalid route parameter, unsupported value, or business rule failure.
401 Authentication missing or expired. Some auth flows also use this for deliberately generic credential failures.
403 Authenticated but not allowed, compliance block, or invalid token role/scope.
404 Requested record not found. A few legacy routes also use it for missing input, so check route-specific notes.
409 Conflict: duplicate resource, order already queued, or current order state does not allow the requested action.
429 Rate limit hit. Envelope is { "status": "fail", "message": "Too many requests, please try again later." }
500 Server/config/database/internal failure. Global handler hides most internal details and usually returns An internal server error occurred.
503 Service temporarily unavailable, mainly used during checkout if provider/admin balance conditions are not satisfied.

POST /api/stripe/webhook

Stripe webhook endpoint. Verifies signature with raw request body, updates payment/order state, marks promo usage, records tax details, and queues eSIM provisioning after payment succeeds.

Key files

  • src/app.js
  • stripe/stripeWebhook.js
  • utils/Queues/purchaseQueueModel.js
  • services/type3TransientPayload.js

Request notes

Must receive Stripe raw JSON body. Do not send normal parsed JSON from manual clients unless you are reproducing Stripe exactly.

Typical success response

{
  "received": true
}
400 Webhook verification failed means the Stripe signature or raw body is invalid.
Side effect note This route can silently accept already processed or partially irrelevant events and still return { received: true } for idempotency.

Known consistency gaps the frontend should account for

  • Some routes return numeric status, some return string status, and some use success.
  • POST /api/auth/otp/resend returns 404 for missing email, which is unusual.
  • POST /api/auth/otp/verify returns 401 for missing fields, not 400.
  • POST /api/admin/delete/sim-multiplier returns 400 when the row is not found, not 404.
  • DELETE /api/admin/promocode/:id currently returns success without explicitly checking affected rows.
